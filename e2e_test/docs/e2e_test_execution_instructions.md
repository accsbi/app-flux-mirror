# E2Eテスト実施指示書

## 1. 目的と対象

### 1.1 目的

Playwrightを使用し、対象アプリケーションのE2Eテストを包括的に実施・ドキュメント化すること。

> ⚠️ **重要:** Playwright MCPによる一時的なブラウザ操作だけで完了としてはいけません。必ずテストケースの作成、コード実装、実行、証跡保存を行ってください。

### 1.2 対象URL

* `http://localhost:3000/`（※プロジェクト設定や既存ドキュメントに応じて適宜修正すること）

### 1.3 対象範囲

* **テスト対象:** 初期表示（ローディング、タイトル、エラーなし）、メニュー画面と全ボタン、画面遷移、モーダル・ダイアログ（開閉・背景操作）、言語切り替え、設定変更、ゲーム開始までの疎通、外部リンク、エラー表示（Console/Network）、レスResponsive表示（`360×800`, `412×915`）
* **対象外:** ゲーム内部の細かい勝敗判定や、複雑なゲームロジック（別途指示がない限り）

---

## 2. 必須作業手順

手順を省略したり、順序を入れ替えたりしてはいけません。

```
【手順1】既存資料の確認（CLAUDE.md, README.md, 既存テスト等の確認）
    ▼
【手順2】アプリケーションの起動確認（起動失敗時はコマンド・エラー・原因を記録）
    ▼
【手順3】画面と操作要素の洗い出し（Playwright MCP/CLIを用い、実画面の要素を網羅）
    ▼
【手順4】テストケース一覧の作成 と テストコードの実装・実行
    ▼
【手順5】テスト結果の文書化 と 最終報告

```

## 3. テストケース一覧の作成規則

### 3.1 記載フォーマット

各ケースに一意のID（例：`MENU-001`, `GAME-001`）を付与し、重複なく記載してください。

| ID | 対象画面 | 前提条件 | 操作 | 期待結果 | 証跡 |
| --- | --- | --- | --- | --- | --- |
| MENU-001 | メニュー | アプリ起動済み | メニュー画面を表示する | 主要ボタンがすべて表示される | [原寸表示]("相対パス")<br>

 |

### 3.2 証跡（スクリーンショット）列のルール

* **必須構成:** `[原寸表示](相対パス)<br><img src="相対パス" alt="ID" width="160">` の形式で、原寸リンクとサムネイル（通常 `width="160"`、縦長スマホ用は `width="120"`）を必ずセットで記述する。
* **複数画像・ビューポート:** 証跡列の中に `<br><br>` を挟んで縦に並べる。
* **禁止事項:** Base64形式での埋め込みは禁止。存在しない画像へのリンクは不可（取得不可時は理由を明記）。
* **保存先:** `e2e_test/<game>/screenshots/`

## 4. Playwright テストコード実装規則

### 4.1 構造と配置

* **作成先:** `/home/dev/wsl_pj/playing_cards/e2e_test/<game>/specs/` 内（例：`menu.spec.ts`, `responsive.spec.ts`）
* **独立性:** 各テストは単独で実行可能とし、前のテストの状態に依存させない（`beforeEach` 等を活用）。

### 4.2 コード記述ルール

* **テスト名:** 必ず対応するテストケースIDを含める。
```ts
test('MENU-001 メニュー画面の主要ボタンが表示される', async ({ page }) => { ... });

```


* **アサーション:** 操作だけで終わらせず、必ず1つ以上の `expect()` を記述する。
* **Locatorの優先度:** `getByRole` ＞ `getByLabel` ＞ `getByPlaceholder` ＞ `getByText` ＞ `getByTestId` の順。CSSの階層構造（例：`div > div > button`）に依存する指定は避ける。
* **待機処理:** `page.waitForTimeout(3000)` などの固定時間待機は原則禁止。状態変化（`toBeVisible()` 等）を待つ。※アニメーション等で必要な場合は理由をコメントに残すこと。

## 6. 失敗時のエラー分類

テストが失敗した場合は、原因を以下の4つに分類して記録してください。

1. **`APPLICATION_BUG`**: アプリケーション側の不具合（動かない、画面崩れ、Consoleエラー等）。
2. **`TEST_BUG`**: テストコード側の不具合（Locatorの間違い、待機条件ミスなど）。**※これのみを修正対象とする。**
3. **`ENVIRONMENT_ERROR`**: 環境起因（サーバー未起動、ポート衝突、外部接続不可など）。
4. **`REQUIREMENT_UNCLEAR`**: 仕様不明確（期待される遷移先や挙動が定義されていない状態）。

## 7. 禁止事項と完了条件

### 7.1 禁止事項

* テストケース一覧を作らずにテストを始める、またはコードを作らずに終了する。
* `expect()` のないテストを大量に作る。
* 失敗したテストを削除・`skip`・コメントアウトして隠蔽する。
* `test.only` を残したままにする。
* アプリの不具合を通すために、期待結果やテストコード、アプリコードを無断で変更する。
* 「問題ありません」の一文だけで報告を済ませる。

### 7.2 完了条件（チェックリスト）

* [ ] 対象画面・操作要素を実際にすべて洗い出した
* [ ] テストケース一覧（ID付き）を作成し、サムネイルのリンク切れがないことを確認した
* [ ] アサーションを含むPlaywrightテストコードを作成・実行した
* [ ] HTMLレポート、失敗時のTrace、動画、必要なスクリーンショットを保存した
* [ ] 全テストケースIDとテストコード内のIDが1対1で対応していることを確認した
* [ ] `test.only` や理由のない `skip` が残っていない
* [ ] 失敗原因を分類し、未実施項目があれば理由を明記した
* [ ] テスト結果文書（`playwright-report.html`）を作成した
