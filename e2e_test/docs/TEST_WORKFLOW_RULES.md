# テスト運用ルール（必読・徹底）

このプロジェクトの **UIテスト**の進め方。AI（および作業者）は必ずこれに従う。
※ここは「テスト」のルール。**INFO/ストア掲載画像とは別タスク**。

---

## 0. テスト と INFO画像 は別タスク（混同禁止）

| | テスト画像 | INFO/ストア画像 |
|---|---|---|
| 形式 | **JPG（quality~80・軽量）** | **PNG** |
| 解像度 | **実機スマホ解像度（例 1081×2340 等）。1080×1920 にしない** | **1080×1920 固定（規格）** |
| 用途 | 検証の証跡（**Google には公開しない**） | Google Play 等に**公開** |
| 置き場所 | `e2e_test/<app>/img/` | `catalog/google_play_store_images/...` |

- **形式（JPG/PNG）だけでも区別できる。**
- 例外：**急ぎの簡単テスト**のときだけ 1080×1920 を使う場合がある（特例）。
- テストと INFO を**一緒に作るのは「別途の指示」があるときだけ**。通常は別々。

---

## 1. 【絶対禁止】ユーザーに作業をさせない

- 「**スクショをお願いします**」「**テストしてください**」の類を**絶対に言わない**（過去何度も指摘）。
- ユーザーは**基本デバッグしない**。テスト・スクショ・確証は**全部 AI が自分でやる**（Playwright 等）。
- デバッグが要っても、ここではユーザーにやらせない（総合的に別途行う前提）。

---

## 2. 【スクショ欄 NULL 禁止】

- CSV の「スクショ」列を**空にしない**。**OK / NG / 対象外 すべて必ずファイルパスを入れる**。
- ユーザーは画面を見ないと良し悪しが分からない。**NG こそスクショで「どこが悪いか」を残す**。
- エラー・クラッシュログ画面・黒画面でも**スクショは撮れる**＝NULL の言い訳にしない。

---

## 3. 運用フロー（CSV → 連番HTML → 修正 → 再テスト）

1. テスト表は **CSV が単一ソース**：`e2e_test/<app>/test-cases*.csv`。スクショ列＝**ファイルパス**。
2. CSV が完成したら **`convert_html.py` を叩く** → HTML 生成。
   - **必ず連番で保存・上書き禁止（日付はつけない）**：`test-cases-menu_001.html` → 次は `_002` …。
   - 基の CSV は `test-cases-menu.csv`。PY 側で既存なら連番で別名保存する実装。
   - ユーザーはこの**連番 HTML** を見る。
3. **NG を見つけたら**：AI が対処方法を考え → 実装 → **再テスト**。
   - OK になったら **CSV を上書きで反映**（**既に OK の行は触らない・NG を OK にするだけ**）。**スクショも更新**。
4. 再度 `convert_html.py` を叩いて **連番つき HTML**（`_002`, `_003` …）。

---

## 4. UI 寸法・レイアウト（判定基準）

- メニュー等は **540×960 基準デザインを `applyStageScale` で縮小して収める**（`docs/UI_TEST_CASES.md` LAY02「ステージは縮小スケールで収まり、はみ出し・見切れなし」）。
- **UI 寸法は 8 の倍数**・**操作ボタン縦幅 ≥88px（スマホ基準）**・**豆粒/見切れ/はみ出し禁止**。
- **vh 等の半端値で個別に縮小しない**（CLAUDE.md §35「半端値禁止」）。固定px非レスポンシブは違反。
- **WEB と render 比較してから直す**（独自実装に置き換えない）。

---

## 5. 変換器

- `scripts/convert_html.py` の **1本だけ**（タイポ `convert_htmt` は作らない）。
- 実行：`convert_html <file.csv>`（PATH ラッパー）または `uv run python scripts/convert_html.py -- <file.csv>`。
- スクショは Pillow で 600px 縮小し base64 埋め込み（どのビューアでも表示）。判定列 OK=緑/NG=赤/対象外=橙。
